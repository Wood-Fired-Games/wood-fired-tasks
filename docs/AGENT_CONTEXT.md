# Agent Context Contract

Owner: Repository maintainers
Status: Authoritative contract. Defines structure; does not document the project.

## 1. Mission

Any AI coding agent, from any vendor, must be able to do useful work on a freshly cloned `wood-fired-tasks` repository using only files committed to that repo. No chat transcript, no task tracker, no proprietary memory, no vendor account, and no local machine context are required. This document is the contract the rest of the project (tasks #273–#286) implements; it specifies what files exist, what they must contain, how big they may be, who owns them, and how they stay fresh.

## 2. Canonical files and read order

The table below lists every agent-facing file the repository commits to ship. Read order is the recommended ingest sequence for a cold start. Size budgets are upper bounds; smaller is better. Tokens are estimated at ~4 chars per token.

### 2.1 Decisions

- **Machine manifest path: `.agent-context.json` at repo root.** Rationale: a root-level dotfile is the lowest-friction discovery target for any tool that scans repo roots (mirrors `.editorconfig`, `.nvmrc`). It is not nested under `docs/` because consumers are tools, not humans.
- **Navigation index: single `docs/NAVIGATION.md`.** Rationale: one file is easier to keep fresh than per-surface indexes, and a single read pulls all five surface recipes (API, MCP, CLI, DB, Slack) into context with one I/O.

## 3. Authority taxonomy

Every canonical file falls into exactly one bucket.

### 3.1 Authoritative

Hand-written, reviewed in PRs, the source of truth. Agents may quote these as ground truth.

- `AGENTS.md`
- `docs/AGENT_CONTEXT.md` (this file)
- `docs/REPO_MAP.md`
- `docs/ARCHITECTURE.md`
- `docs/WORKFLOWS.md`
- `docs/NAVIGATION.md`
- `docs/API.md`, `docs/MCP.md`, `docs/CLI.md`, `docs/SETUP.md`, `docs/SLACK.md`, `docs/RELEASE.md`
- `CONTRIBUTING.md`, `README.md`

### 3.2 Generated

Produced by a committed script from code or schema. Humans edit the script, not the artifact. Each generated file carries the header:

```
<!-- generated by scripts/<name>.ts; do not edit -->
```

- `.agent-context.json` (generated from a committed script; see task #275)
- `docs/INTERFACES.md` (generated from route, MCP-tool, and CLI-command registries; see task #279)

### 3.3 Compatibility / adapter

Vendor-specific files that exist only to point a vendor toolchain at the authoritative content. They MUST NOT carry unique project facts.

- `CLAUDE.md` (if added) — body: `> See [AGENTS.md](AGENTS.md).` plus optional vendor-only config notes.
- `llms.txt` (community LLM site-map convention) — pointer or compact link list only.
- `.cursor/rules/*` (if added) — pointer or vendor-config only.
- `.gemini/*`, `.codex/*` (if added) — pointer or vendor-config only.
- `.claude/commands/*` — Claude-specific slash commands. Allowed because they are vendor config, not project facts.

If a fact lives only in a compatibility file, that fact is in the wrong place. Move it into the authoritative tier and replace the compatibility file with a pointer.

## 4. Size and content budgets

### 4.1 Per-file budgets

See section 2 table. The hard limits there are upper bounds. Targets:

- `AGENTS.md`: ≤ 150 lines. Anything longer belongs in `docs/`. This is the "first thousand tokens of an agent's context" file and must stay scannable.
- `docs/AGENT_CONTEXT.md`: ≤ 400 lines.
- `docs/REPO_MAP.md`: ≤ 250 lines.
- `docs/ARCHITECTURE.md`: ≤ 300 lines.
- `docs/WORKFLOWS.md`: ≤ 250 lines.
- `docs/INTERFACES.md`: ≤ 400 lines.
- `docs/NAVIGATION.md`: ≤ 300 lines.

### 4.2 Minimum useful context

The intended layered load is:

1. **First load (≤ ~3k tokens):** `AGENTS.md` only. Sufficient to identify which deep doc to open next.
2. **Second load (≤ ~10k tokens cumulative):** `AGENTS.md` + `docs/REPO_MAP.md` + `docs/ARCHITECTURE.md` + `docs/WORKFLOWS.md`. Sufficient to run build/test/lint and reason about boundaries.
3. **Drill-down:** `docs/INTERFACES.md` + `docs/NAVIGATION.md` to find the right surface, then a single deep doc (`docs/API.md`, `docs/MCP.md`, `docs/CLI.md`, `docs/SLACK.md`) on demand.

### 4.3 Overflow rule

Any agent-facing doc that exceeds its line budget MUST be split. Split by linking out to an existing or new deep doc, not by silently growing the budget. The budgets in section 2 are enforced by the freshness check in task #281.

### 4.4 Forbidden content (everywhere agent-facing)

- Secrets, API keys, tokens, passwords, signing keys.
- Local absolute paths (`/home/...`, `/Users/...`, `C:\\...`).
- Per-developer machine state (shell aliases, personal tool paths).
- Personal email addresses or contact info.
- Generated stack traces, raw log dumps, or transient debug output.

## 5. Freshness and ownership rules

### 5.1 Ownership header

Every authoritative file starts with an `Owner:` line within the first three lines of the file. The value is a short role string (e.g. `Owner: API maintainers`, `Owner: MCP maintainers`, `Owner: Repository maintainers`). Per-person ownership is out of scope; roles are the unit.

### 5.2 PR-time freshness rule

A PR that modifies any surface named in `docs/INTERFACES.md` MUST update the matching agent-facing doc in the same PR. Specifically:

- Changes under `src/api/routes/**` → update `docs/INTERFACES.md` (regen) and `docs/API.md` if behavior changes.
- Changes under `src/mcp/tools/**` → update `docs/INTERFACES.md` (regen) and `docs/MCP.md` if behavior changes.
- Changes under `src/cli/commands/**` → update `docs/INTERFACES.md` (regen) and `docs/CLI.md` if behavior changes.
- Changes under `src/db/migrations/**` → update `docs/ARCHITECTURE.md` data-flow section if the schema boundary moves.
- Changes under `src/slack/**` → update `docs/SLACK.md` if surface changes.

### 5.3 CI freshness check

A CI job (task #281) MUST:

- Re-run the generator scripts for every generated file in section 3.2 and fail if the working tree is dirty afterwards.
- Verify each authoritative file in section 3.1 is within its section 4.1 line budget.
- Verify each authoritative file has an `Owner:` line in the first three lines.
- Scan agent-facing files for the forbidden content patterns in section 4.4.

Until task #281 lands, this rule is documented but not enforced. Adding the CI check is a required follow-up.

### 5.4 Generated-file header

Every file in section 3.2 starts with the literal header:

```
<!-- generated by scripts/<name>.ts; do not edit -->
```

(or `// generated ...` for JSON, using the JSON-compatible variant chosen by task #275 — the manifest itself is JSON, so the header lives in a sibling `.agent-context.json.meta` or a `_generated` field inside the JSON object; task #275 picks one and documents it.)

## 6. Boundary: vendor-neutral vs. vendor-specific

### 6.1 Vendor-neutral surface

The only authoritative location for project-specific knowledge is:

- `AGENTS.md`
- `docs/**/*.md`
- `.agent-context.json`
- `CONTRIBUTING.md`, `README.md`

These files MUST NOT mention any specific AI vendor by name in normative text. They MAY mention vendors in non-normative examples (e.g. "if your tool reads `AGENTS.md`...").

### 6.2 Vendor-specific surface

The following directories and files MAY exist for vendor-quality-of-life:

- `.claude/` — Claude Code config: slash commands, hooks, MCP client setup.
- `CLAUDE.md` — Claude-specific entry point.
- `.cursor/` — Cursor rules.
- `.codex/` — OpenAI Codex config.
- `.gemini/` — Gemini config.
- Any future `.<vendor>/` directory.

Rules for the vendor-specific surface:

1. Every vendor-specific entry-point document (e.g. `CLAUDE.md`, `.cursor/rules/*.md`) MUST be either:
   - A thin pointer whose entire body is `> See [AGENTS.md](AGENTS.md).` plus optional vendor-config notes; or
   - Vendor configuration only (slash commands, MCP client wiring, tool allow-lists).
2. A vendor-specific file MUST NOT contain unique project facts. If you find yourself writing one, move the content to the authoritative tier and replace the vendor file with a pointer.
3. Removing all vendor-specific files MUST leave the project fully usable by any agent that reads `AGENTS.md`.

### 6.3 Forbidden in both tiers

The section 4.4 forbidden-content list applies everywhere: vendor-neutral and vendor-specific.

## 7. Definition of done (project 12)

Project 12 is done when this acceptance criterion holds:

> A fresh agent can identify the right files, commands, tests, and constraints for representative API, MCP, CLI, DB, Slack, and docs changes using only committed repo files.

The future agent-onboarding smoke test (task #282) MUST cover the five probe scenarios below. Each probe begins from a fresh clone with no prior context; "first N reads" means the first N file reads the agent issues after reading `AGENTS.md`.

### 7.1 Probe A — API change

Prompt: "Add a new REST endpoint `GET /tasks/:id/history`."
Pass condition: within the first 3 reads after `AGENTS.md`, the agent opens `docs/INTERFACES.md`, `docs/API.md`, and a file under `src/api/routes/`. Within the first 5 reads it locates an existing route test under `src/api/__tests__/` to use as a pattern.

### 7.2 Probe B — MCP change

Prompt: "Add a new MCP tool `archive_task`."
Pass condition: within the first 3 reads after `AGENTS.md`, the agent opens `docs/MCP.md`, a file under `src/mcp/tools/`, and the MCP test pattern under `src/mcp/__tests__/`.

### 7.3 Probe C — CLI change

Prompt: "Add a new CLI subcommand `bugs export`."
Pass condition: within the first 3 reads after `AGENTS.md`, the agent opens `docs/CLI.md`, a file under `src/cli/commands/`, and the CLI test pattern under `src/cli/__tests__/`.

### 7.4 Probe D — DB change

Prompt: "Add a migration that adds a `priority` column to `tasks`."
Pass condition: within the first 3 reads after `AGENTS.md`, the agent opens `docs/ARCHITECTURE.md`, a file under `src/db/migrations/`, and `src/db/migrate.ts`. Within the first 5 reads it identifies the migration test pattern under `src/db/__tests__/`.

### 7.5 Probe E — Slack / docs change

Prompt: "Add a Slack slash command response for `/bugs status`."
Pass condition: within the first 3 reads after `AGENTS.md`, the agent opens `docs/SLACK.md`, a file under `src/slack/`, and locates `slack-app-manifest.yml` at the repo root.

### 7.6 Project-level pass criterion

All five probes pass without the agent reading any file outside the committed repo, without consulting any task tracker, and without any chat history beyond the probe prompt itself.

## 8. Recommended read paths

One mini-table per intent. Each path is capped at the smallest useful set of files (≤ 5). Files marked `(coming soon)` are reserved by the contract but not yet on disk; skip them if absent.

| Intent | Read in order |
|---|---|
| API change (REST routes) | `AGENTS.md` → `docs/AGENT_CONTEXT.md` → `docs/API.md` → `src/api/routes/<resource>/` → `src/api/__tests__/` |
| MCP tool change | `AGENTS.md` → `docs/MCP.md` → `src/mcp/tools/` → `src/mcp/__tests__/` → `docs/INTERFACES.md` (coming soon) |
| CLI subcommand change | `AGENTS.md` → `docs/CLI.md` → `src/cli/commands/` → `src/cli/__tests__/` |
| DB schema / migration | `AGENTS.md` → `src/schemas/` → `src/db/migrations/` → `src/db/migrate.ts` → `src/db/__tests__/` |
| Slack change | `AGENTS.md` → `docs/SLACK.md` → `src/slack/` → `slack-app-manifest.yml` |
| Test-only fix | failing test file → unit under test → `vitest.config.ts` → `src/<surface>/__tests__/` (sibling pattern) |
| Docs / agent-context change | `docs/AGENT_CONTEXT.md` → `AGENTS.md` → `.agent-context.json` → `scripts/agent-context/manifest.ts` |
| Release / packaging change | `docs/RELEASE.md` → `CHANGELOG.md` → `package.json` → `.github/workflows/` |

When `docs/NAVIGATION.md` lands it will supersede this table for finer-grained, per-surface recipes; the table above remains the minimum-useful map.

## 9. Glossary of repo-specific terms

Repo-specific terms. Definitions are extractive (read the source if you need more); one line each.

| Term | Definition |
|---|---|
| **task** | Unit of work tracked in the `tasks` table; owns title, status, project_id, optional parent_task_id, assignee, tags. |
| **project** | Container for tasks (`projects` table). A task belongs to exactly one project. |
| **subtask** | A task with a non-null `parent_task_id`; parent auto-completes when all subtasks are done. |
| **dependency** | A directed `task → task` edge in `task_dependencies`; blocks dependent until predecessor is done. |
| **comment** | Free-text annotation on a task (`comments` table); append-only from the API surface. |
| **claim** | Atomic acquire+release of a task via the claim endpoint; uses optimistic locking and the 24h idempotency cache. |
| **status** | One of six legal values: `open`, `in_progress`, `blocked`, `done`, `closed`, `backlogged`. |
| **local MCP** | In-process stdio MCP server (`npm run mcp:dev`); talks straight to the shared SQLite file. |
| **remote MCP** | HTTP MCP server (`npm run mcp:remote`); proxies tool calls into the REST API over the network. |
| **service** | Shared business-logic layer under `src/services/`; every surface (API, MCP, CLI, Slack) calls services, never repositories. |
| **repository** | SQL access layer under `src/repositories/`; the only code that talks to better-sqlite3 directly. |
| **schema** | Zod schemas in `src/schemas/`; the single validation source of truth for every surface. |
| **migration** | A file under `src/db/migrations/` run by umzug; named with a monotonic numeric prefix. |
| **event** | In-process event-bus payload; types listed in `src/events/types.ts` (e.g. `task.created`, `task.status_changed`, `project.deleted`). |
| **WorkflowEngine** | Service that cascades parent task completion and auto-unblocks dependents when their predecessors finish. |
| **idempotency** | 24h replay cache keyed by `Idempotency-Key` header; applied today only on the claim endpoint. |
| **SSE** | Server-Sent Events stream at `GET /api/v1/events`; long-lived per-key/per-IP connection capped by `SSE_MAX_CONNECTIONS_*`. |
| **PAT** | Personal access token (`wft_pat_…`). The CLI reads it from `API_KEY` and the remote MCP server from `WFT_API_KEY`; both send it as `Authorization: Bearer <pat>`. The legacy `X-API-Key` shared-secret header was **removed in v2.0** (and `API_KEYS` is no longer an auth method — it only seeds inert legacy rows). |

## 10. Source of truth

This document is the **source of truth for agent onboarding**. Vendor-specific files (`CLAUDE.md`, `.cursor/`, `.gemini/`, `.codex/`, any future `.<vendor>/`) are **adapters only** — they MUST point back here and MUST NOT carry unique project facts. Any divergence between this contract and the on-disk files (missing canonical file, oversized doc, vendor file with unique facts, stale read paths) is a freshness bug worth filing. See §6 for the full vendor-neutrality boundary rules.

## 11. Relationship to existing docs

This contract does not rewrite or supersede the existing deep docs. `docs/API.md`, `docs/MCP.md`, `docs/CLI.md`, `docs/SETUP.md`, `docs/SLACK.md`, `docs/RELEASE.md`, `docs/CODE_QUALITY_ROADMAP.md`, `CONTRIBUTING.md`, and `README.md` remain authoritative for their topics. The new agent-facing files (`AGENTS.md`, `docs/REPO_MAP.md`, `docs/ARCHITECTURE.md`, `docs/WORKFLOWS.md`, `docs/INTERFACES.md`, `docs/NAVIGATION.md`, `.agent-context.json`) add a compact, navigable top layer; they link down to the deep docs rather than duplicate them.

## 12. Change control

Changes to this contract require:

1. A PR that updates `docs/AGENT_CONTEXT.md`.
2. A corresponding update to `AGENTS.md` if the canonical file list, read order, or budgets change.
3. A corresponding update to the CI freshness check (task #281) if the rules in section 5 change.

Adding a new canonical file requires adding a row to the section 2 table, classifying it in section 3, and giving it a size budget in section 4.
