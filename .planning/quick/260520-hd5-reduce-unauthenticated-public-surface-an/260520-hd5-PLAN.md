# Quick Task 260520-hd5: Reduce unauthenticated public surface and bound long-lived connections

**Date:** 2026-05-20
**Mode:** quick-full

## Description

Three exposure-point hardening fixes before open-sourcing the bug tracker:

1. **Swagger gating in production.** `/docs` and `/docs/json` are currently registered globally before `/api/v1` auth, so they leak the full API surface to anyone who can reach the port. In `NODE_ENV=production` they must be disabled unless an operator explicitly sets `ENABLE_SWAGGER_IN_PRODUCTION=true`, and in that case must require the same X-API-Key auth as `/api/v1`.

2. **Minimal default /health.** `/health` is intentionally public for k8s/load-balancer probes but currently returns SSE client count, uptime, and per-component status — useful information for an attacker mapping the deployment. Default `/health` to a minimal status (`{ status, timestamp, version }`), and add `/health/detailed` (gated by the same canonical auth plugin) for ops monitoring.

3. **SSE connection caps.** The authenticated SSE endpoint has only a 10-minute max age and no concurrent-connection ceiling. One leaked key → one attacker can open thousands of connections and exhaust the server. Add per-key, per-IP, and global caps; reject excess with 429 + `Retry-After`.

### Task 3: SSE connection caps (per-key, per-IP, global)

**Files:**
- `src/config/env.ts`
- `src/events/sse-manager.ts`
- `src/api/routes/events.ts`
- `src/events/__tests__/sse-manager.test.ts` (update — touch existing test surface only if API requires it)
- `src/api/__tests__/sse-caps.test.ts` (new)

**Action:**
1. Extend `configSchema` in `src/config/env.ts`:
   ```ts
   SSE_MAX_CONNECTIONS_PER_KEY: z.string().min(1).default('4').transform(Number),
   SSE_MAX_CONNECTIONS_PER_IP:  z.string().min(1).default('8').transform(Number),
   SSE_MAX_CONNECTIONS:         z.string().min(1).default('200').transform(Number),
   ```
2. Extend `SSEManager` constructor to accept optional `maxPerKey`, `maxPerIp`, `maxTotal` (defaults match config). Extend `SSEConnection` interface with `apiKey: string` and `ip: string`. Update `addConnection(id, reply, filters, lastEventId?, meta: { apiKey, ip })` signature.
3. Add public method `SSEManager.canAccept(apiKey: string, ip: string): { ok: true } | { ok: false; reason: 'per-key'|'per-ip'|'global'; retryAfterSeconds: number }`. Reason maps to the violated cap; `retryAfterSeconds` is a constant 30.
4. Add private counts: derive from `this.connections.values()` (O(n), n is bounded by `maxTotal` <= a few hundred) — no separate index needed at this scale, and avoids drift bugs when connections close out-of-band.
5. In `src/api/routes/events.ts`:
   - Read `apiKey = request.headers['x-api-key'] as string` (auth already validated). `ip = request.ip`.
   - Call `server.sseManager.canAccept(apiKey, ip)`. If `ok: false`, set header `Retry-After: 30`, send `{ error: 'TOO_MANY_CONNECTIONS', message: 'Too many SSE connections (limit: <reason>). Retry in 30 seconds.' }` with status 429.
   - Otherwise proceed to `addConnection` with the meta.
6. Wire `createServer` to pass config values to the `SSEManager` constructor.

**Verify:**
- With `SSE_MAX_CONNECTIONS_PER_KEY=2`, opening 3 connections with same key → 3rd returns 429 with `Retry-After: 30`.
- With `SSE_MAX_CONNECTIONS_PER_IP=2`, opening 3 connections from same IP but different keys → 3rd returns 429.
- With `SSE_MAX_CONNECTIONS=2`, opening 3 connections (any keys/IPs) → 3rd returns 429.
- Closing one connection (raw 'close' event) drops the count → a new connection succeeds.
- Existing `events.test.ts` continues to pass.

**Done:**
- New `sse-caps.test.ts` covers per-key, per-IP, global cap, and post-close-recovery scenarios using direct `SSEManager` API (`reply.inject` is not viable for SSE — use `SSEManager.canAccept` and `addConnection` directly with mock replies, mirroring the existing `sse-manager.test.ts` pattern).
- Route-level test in same file uses `server.inject` to verify the 429 + Retry-After response shape end-to-end.
