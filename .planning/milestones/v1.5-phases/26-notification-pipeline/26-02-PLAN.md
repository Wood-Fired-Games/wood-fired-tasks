**1. Create `src/slack/notifier.ts`:**

```typescript
import type { WebClient } from '@slack/web-api';
import type { KnownBlock } from '@slack/types';
import type { Logger } from 'pino';
import { eventBus } from '../events/event-bus.js';
import type { TaskEvent } from '../events/types.js';
import type { SlackChannelSubscriptionRepository } from './repositories/channel-subscription.repository.js';
import type { ProjectService } from '../services/project.service.js';
import { formatTaskNotification } from './task-formatter.js';

export class SlackNotifier { ... }
```

Constructor takes: `client: WebClient`, `subscriptionRepo: SlackChannelSubscriptionRepository`, `projectService: ProjectService`, `logger: Logger`.

**`start()` method:**
Subscribe to these 5 task event types: `task.created`, `task.updated`, `task.status_changed`, `task.claimed`, `task.deleted`. For each, register a SYNCHRONOUS handler that calls `this.handleTaskEvent(eventType, event).catch(err => this.logger.error({ err, eventType }, 'SlackNotifier: unhandled error'))`. Store the unsubscribe functions in a private `unsubscribes: Array<() => void>` array.

CRITICAL: The handler registered with eventBus.subscribe MUST be synchronous (not async). It calls the async method and chains `.catch()`. This prevents unhandled promise rejections because EventBus wraps handlers in try/catch which only catches synchronous throws.

**`stop()` method:**
Call each unsubscribe function and clear the array.

**`private async handleTaskEvent(eventType: string, event: TaskEvent)` method:**
1. Extract `projectId` from `event.data.project_id`
2. Call `this.subscriptionRepo.findSubscribedChannels(projectId, eventType)` — this is synchronous (better-sqlite3) so it's fine inline
3. If no channels, return early
4. Resolve project name: `try { const project = this.projectService.getProject(projectId); projectName = project.name; } catch { projectName = 'Project #' + projectId; }`
5. Call `formatTaskNotification(event, projectName)` to get Block Kit blocks
6. Create fallback text: `${eventType}: ${event.data.title}`
7. Use `Promise.allSettled()` to post to each channel independently via `this.postWithRetry(channelId, blocks, fallbackText)`
8. Log any rejected results at error level with channel ID and event type

**`private async postWithRetry(channelId: string, blocks: KnownBlock[], text: string, maxRetries = 2)` method:**
Loop from attempt 0 to maxRetries:
- Try `await this.client.chat.postMessage({ channel: channelId, blocks, text })`
- On success, return
- On error: check if permanent (`err.data?.error === 'not_in_channel'` or `'channel_not_found'` or `'invalid_auth'` or `'token_revoked'`) — if permanent, throw immediately (no retry)
- If transient and not last attempt: `await new Promise(r => setTimeout(r, 500 * (attempt + 1)))` then continue
- If last attempt: throw

IMPORTANT: The existing `onClose` hook (lines 144-150) already handles `slackService.stop()`. The new `onClose` hook for `slackNotifier.stop()` is additive — Fastify supports multiple onClose hooks and executes them all. The notifier's stop() MUST be called BEFORE `slackService.stop()` (which disconnects the WebSocket). Since Fastify executes onClose hooks in registration order, and the slackService onClose is registered at line 144 while the notifier onClose is registered later inside the `if (slackApp)` block, the notifier's stop will run AFTER slackService.stop. This is acceptable because `slackNotifier.stop()` only unsubscribes from EventBus — it does not make Slack API calls. The synchronous unsubscribe prevents any NEW events from being processed after shutdown begins.

Actually, review the hook ordering more carefully. The existing onClose at line 144 runs `await slackService.stop()`. If we add a new onClose later, it runs after. But `slackNotifier.stop()` is purely synchronous (removes EventBus listeners) — it does not need the Slack connection. So ordering is fine either way.
