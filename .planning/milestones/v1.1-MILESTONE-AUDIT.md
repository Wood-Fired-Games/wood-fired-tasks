## Known Issues (Non-Blocking)

### 1. Phase 7 Verification dotenv Gap -- RESOLVED

The Phase 7 verification report flagged stdout contamination from `dotenv.config()` lacking `{quiet: true}`. This has been fixed:
- `src/cli/config/env.ts` line 12 now reads: `dotenv.config({ path: envPath, quiet: true })`

### 2. Error Handler Does Not Use JSON Envelope in JSON Mode

`src/cli/output/error-handler.ts` always uses `console.error()` (stderr) for errors, even in `--json` mode. It does not output a `{ success: false, error: {...} }` envelope to stdout. This means:
- In JSON mode, errors appear only on stderr (not on stdout)
- A piping consumer (`jq`) will see no output on stdout when errors occur

This is acceptable behavior -- errors on stderr is the POSIX convention, and the `jsonError()` utility exists for future enhancement if needed. Not a v1.1 blocker.

### 3. comment-delete Takes Two Arguments

`CMD-13` requirement says `tasks comment-delete <id>` but implementation takes `<task-id> <comment-id>`. This matches the REST API's nested route structure and is the correct design. The requirement wording is ambiguous but the implementation is functionally complete.
