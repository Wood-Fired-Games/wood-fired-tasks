   ```typescript

   ```

2. Add `backlogged` entry to `VALID_STATUS_TRANSITIONS` and add `'backlogged'` to `open`'s transition list:
   ```typescript
   export const VALID_STATUS_TRANSITIONS: Record<TaskStatus, TaskStatus[]> = {
     open: ['in_progress', 'blocked', 'closed', 'backlogged'],
     in_progress: ['done', 'blocked', 'open'],
     blocked: ['open', 'in_progress'],
     done: ['closed', 'open'],
     closed: ['open'],
     backlogged: ['open'],
   };
   ```

   Design rationale: `backlogged -> open` is the ONLY valid transition from backlogged. You cannot go `backlogged -> in_progress` directly -- this enforces the triage workflow where backlogged tasks must be explicitly promoted to open before agents can claim them.

**Part D: Create src/db/migrations/005-backlogged-status.ts**

Create migration file following the exact pattern from the research and from migration 001-initial-schema.ts. This migration uses the SQLite table rebuild pattern because CHECK constraints cannot be modified with ALTER TABLE.

The migration MUST:
1. Use `db.transaction(() => { ... })()` pattern (same as migration 004)
2. `db.pragma('foreign_keys = OFF')` at the start
3. Create `tasks_new` table with ALL current columns (including `parent_task_id` from migration 002, `estimated_minutes` from 003, `version` and `claimed_at` from 004) and the updated CHECK constraint adding `'backlogged'`
4. Copy all data: `INSERT INTO tasks_new SELECT * FROM tasks`
5. Drop the 3 FTS triggers BEFORE dropping the table: `tasks_fts_insert`, `tasks_fts_update`, `tasks_fts_delete`
6. `DROP TABLE tasks`
7. `ALTER TABLE tasks_new RENAME TO tasks`
8. Recreate ALL indexes: `idx_tasks_project_id`, `idx_tasks_project_status_assignee`, `idx_tasks_status_due_date`, `idx_tasks_assignee`, `idx_tasks_parent_id`
9. Recreate ALL 3 FTS triggers (copy exact SQL from migration 001-initial-schema.ts)
10. `db.pragma('foreign_keys = ON')` at the end

The `down()` function should reverse the process: rebuild the table WITHOUT 'backlogged' in the CHECK constraint. First update any rows with status='backlogged' to status='open' before rebuilding.

CRITICAL: The FTS trigger recreation is the most common pitfall. If triggers are not recreated, task creation will work but FTS search will return no results for new tasks. Copy the exact trigger SQL from migration 001.

The complete column list for tasks_new (in order):
- id INTEGER PRIMARY KEY AUTOINCREMENT
- title TEXT NOT NULL
- description TEXT
- status TEXT NOT NULL DEFAULT 'open' CHECK(status IN ('open', 'in_progress', 'done', 'closed', 'blocked', 'backlogged'))
- priority TEXT NOT NULL DEFAULT 'medium' CHECK(priority IN ('low', 'medium', 'high', 'urgent'))
- project_id INTEGER NOT NULL REFERENCES projects(id) ON DELETE CASCADE
- parent_task_id INTEGER REFERENCES tasks(id) ON DELETE CASCADE
- assignee TEXT
- created_by TEXT NOT NULL
- due_date TEXT
- created_at TEXT NOT NULL DEFAULT (datetime('now'))
- updated_at TEXT NOT NULL DEFAULT (datetime('now'))
- estimated_minutes INTEGER
- version INTEGER NOT NULL DEFAULT 1
- claimed_at TEXT
