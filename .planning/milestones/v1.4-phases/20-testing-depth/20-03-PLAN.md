<objective>
Configure Stryker mutation testing to validate test suite effectiveness by mutating source code and measuring how many mutants are killed by existing tests.

Purpose: Line coverage shows what code is executed during tests but not whether tests actually assert on it. Mutation testing verifies tests detect real bugs by introducing small code changes (mutants) and checking if tests catch them. A high mutation score means the test suite is effective at catching regressions.

Output: `stryker.config.js` configuration, `npm run test:mutation` script, HTML mutation report.
</objective>

  <action>
    1. Install Stryker packages as dev dependencies:
       ```bash
       npm install --save-dev @stryker-mutator/core @stryker-mutator/vitest-runner @stryker-mutator/typescript-checker
       ```

    2. Create `stryker.config.js` at project root with ES module syntax (works because package.json has `"type": "module"`):

       ```javascript
       // @ts-check
       /** @type {import('@stryker-mutator/api/core').PartialStrykerOptions} */
       export default {
         testRunner: 'vitest',
         vitest: {
           configFile: 'vitest.config.ts',
           related: false,
         },
         coverageAnalysis: 'perTest',
         mutate: [
           'src/**/*.ts',
           '!src/**/__tests__/**',
           '!src/**/*.test.ts',
           '!src/db/migrate.ts',
           '!src/cli/bin/tasks.ts',
         ],
         checkers: ['typescript'],
         tsconfigFile: 'tsconfig.json',
         typescriptChecker: {
           prioritizePerformanceOverAccuracy: true,
         },
         reporters: ['html', 'clear-text', 'progress'],
         thresholds: {
           high: 80,
           low: 60,
           break: null,
         },
         packageManager: 'npm',
       };
       ```

       Critical configuration choices:
       - `vitest.related: false` — MUST be false. The project's integration tests use `createTestApp()` factory pattern and do not directly import source files. With `related: true`, Stryker's static import analysis would skip those tests and report false "survived" mutants. See RESEARCH.md Pitfall 1.
       - `checkers: ['typescript']` — Filters type-invalid mutants before running tests. Prevents crashes from mutants that produce valid JS but invalid TS. See RESEARCH.md Pitfall 2.
       - `typescriptChecker.prioritizePerformanceOverAccuracy: true` — Faster mutation runs at cost of occasionally letting through a type-invalid mutant (acceptable tradeoff).
       - `thresholds.break: null` — Do NOT fail on first run. The initial mutation score is unknown. Set a threshold after observing the baseline.
       - Exclusions: `__tests__` dirs, `.test.ts` files, `migrate.ts` (migration runner), `tasks.ts` CLI entry (Commander setup). These are not business logic and should not be mutated.

    3. Add the `test:mutation` script to package.json scripts:
       ```json
       "test:mutation": "stryker run"
       ```

    4. Add Stryker output directories to `.gitignore` if not already present:
       ```
       # Stryker mutation testing reports
       reports/
       .stryker-tmp/
       ```

    5. Run `npm run test:mutation` to verify Stryker executes successfully.

       **Expected behavior:** Stryker will:
       - Discover vitest tests
       - Analyze per-test coverage
       - Generate mutants from `src/**/*.ts` (excluding tests)
       - Filter type-invalid mutants via TypeScript checker
       - Run tests against each mutant
       - Report killed/survived/timeout/no coverage breakdown
       - Generate HTML report in `reports/` directory

       **If Stryker hangs or runs extremely slowly:** Try adding `--concurrency 2` to the stryker run command. The project's `fileParallelism: false` in vitest.config.ts may interact with Stryker's default concurrency.

       **If Stryker reports "no tests found":** Verify vitest.configFile path is correct relative to project root.

       **Note:** The first run may take several minutes (5-15 min depending on hardware) as Stryker tests hundreds of mutants. This is normal. Do not kill the process prematurely.
  </action>
