Integrate EventBus into TaskService and ProjectService to emit domain events after successful database operations.

Purpose: Connect domain logic to event stream. Services emit events AFTER transaction commits to ensure SSE consumers never receive events for failed operations (addresses Pitfall #4: Event Broadcast Race).

NOTE: task.claimed emission deferred to Phase 15 when POST /api/v1/tasks/:id/claim endpoint is implemented. Phase 14 defines the event type but does not emit it.

Output: Services emit typed events for all CRUD operations with entity snapshots in payload.

  <name>Add event emissions to TaskService CRUD operations</name>
  <files>src/services/task.service.ts, src/services/__tests__/task.service.test.ts</files>
  <action>
Modify TaskService methods to emit events AFTER successful repository operations:

1. Import eventBus singleton from src/events/event-bus.ts at top of file

2. In createTask(), after `taskRepo.create()` returns, emit task.created:
   ```typescript
   const task = this.taskRepo.create({ ...result.data, status: 'open' }, result.data.tags);
   eventBus.emit('task.created', {
     eventType: 'task.created',
     timestamp: new Date().toISOString(),
     data: task,
     metadata: { source: 'user' }
   });
   return task;
   ```

3. In updateTask(), after `taskRepo.update()` returns:
   - Always emit task.updated with full entity snapshot
   - If status changed (result.data.status && result.data.status !== existing.status), ALSO emit task.status_changed with { from: existing.status, to: result.data.status } in metadata

4. In deleteTask(), BEFORE `taskRepo.delete()` call:
   - Emit task.deleted with existing task data (last chance to notify before removal)
   - Research rationale: Emit before deletion so consumers can still query related entities

5. Update tests in src/services/__tests__/task.service.test.ts:
   - Mock eventBus.emit using vi.spyOn(eventBus, 'emit')
   - Verify emit called with correct event type and payload structure
   - Verify emit called AFTER repository operation (use vi.mock to track call order if needed)
   - Test that failed operations (ValidationError, NotFoundError) do NOT emit events

Why not emit before database operation: SQLite transactions are synchronous in better-sqlite3. The repository call either succeeds immediately or throws. Emitting after return guarantees transaction committed successfully.

Addresses Pitfall #4: Include full entity snapshot in event payload to prevent 404 race conditions when consumers query API immediately after receiving event.

NOTE: Do NOT emit task.claimed in this plan - claiming functionality does not exist until Phase 15.
