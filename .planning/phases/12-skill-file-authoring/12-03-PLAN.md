- argument-hint: [task-id]

Markdown body (single-tool workflow):
1. Validate task ID from $ARGUMENTS[0] (required, must be positive integer).
2. Call `wood-fired-bugs:update_task` with:

   - updates: { status: 'done' }
3. Confirm completion: "Task <id> marked as done: <title>"

Include valid status transitions to 'done' as reference:
- in_progress -> done (normal completion)
- If task is currently 'open', it can transition to in_progress first, then done. Or open -> in_progress is also valid — the server accepts open -> done if the task was completed without formal pickup.
- If task is 'blocked', it needs to move to open or in_progress first.

Keep under 50 lines total. Use imperative form.

  <action>
Create `skills/tasks/blocked.md`.

YAML frontmatter:
- name: blocked
- description: "Marks a task as blocked and records the blocking reason as a comment. Use when user reports a blocker, dependency issue, or impediment on a task."
- argument-hint: [task-id] [reason]
- disable-model-invocation: false

Markdown body (multi-tool workflow):
1. Parse task ID from $ARGUMENTS[0] (required, positive integer).
2. Parse blocking reason from remaining $ARGUMENTS (required — if not provided, ask user for the reason).
3. Call `wood-fired-bugs:update_task` with:
   - id: task ID
   - updates: { status: 'blocked' }
4. Call `wood-fired-bugs:add_comment` with:
   - task_id: task ID
   - author: 'user'
   - content: "BLOCKED: <reason>"
5. Confirm: "Task <id> marked as blocked. Reason recorded: <reason>"

Include valid status transitions to 'blocked':
- open -> blocked
- in_progress -> blocked

Keep under 60 lines total. Use imperative form.
  </action>
  <verify>File exists at skills/tasks/blocked.md. Has valid frontmatter. Body references both `wood-fired-bugs:update_task` and `wood-fired-bugs:add_comment` (fully qualified). Under 100 lines.</verify>
  <done>skills/tasks/blocked.md exists with correct frontmatter, references update_task (status blocked) and add_comment (reason recording), requires reason from user.</done>
