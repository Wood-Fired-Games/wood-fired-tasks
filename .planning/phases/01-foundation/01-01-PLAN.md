Create the database layer and all shared types.

**src/types/task.ts** -- Define all TypeScript types used across the project:
- `TaskStatus` = `'open' | 'in_progress' | 'done' | 'closed' | 'blocked'`
- `TaskPriority` = `'low' | 'medium' | 'high' | 'urgent'`
- `Task` interface: `id: number`, `title: string`, `description: string | null`, `status: TaskStatus`, `priority: TaskPriority`, `project_id: number`, `assignee: string | null`, `created_by: string`, `due_date: string | null` (ISO8601), `created_at: string`, `updated_at: string`
- `Project` interface: `id: number`, `name: string`, `description: string | null`, `created_at: string`, `updated_at: string`
- `TaskTag` interface: `id: number`, `task_id: number`, `tag: string`
- `CreateTaskDTO`: Omit Task of id, created_at, updated_at -- with description, assignee, due_date optional
- `UpdateTaskDTO`: Partial of title, description, status, priority, assignee, due_date, tags
- `CreateProjectDTO`: `name: string`, `description?: string`
- `TaskFilters` interface: `project_id?: number`, `status?: TaskStatus`, `assignee?: string`, `tags?: string[]`, `due_before?: string`, `due_after?: string`, `search?: string`
- Export the `VALID_STATUS_TRANSITIONS` map: Record<TaskStatus, TaskStatus[]> per the research (open->in_progress/blocked/closed, in_progress->done/blocked/open, blocked->open/in_progress, done->closed/open, closed->open)
- Export `TASK_STATUSES` and `TASK_PRIORITIES` as const arrays for Zod enum usage later.

**src/db/database.ts** -- Database initialization function:
- Export `initDatabase(filepath: string)` that:
  - Creates a new better-sqlite3 Database instance at filepath
  - Sets pragma `journal_mode = WAL`
  - Sets pragma `foreign_keys = ON`
  - Sets pragma `synchronous = NORMAL`
  - Calls `db.timeout(5000)` for busy timeout
  - Returns the Database instance
- Export `initTestDatabase()` that calls `initDatabase(':memory:')` for tests (still sets foreign_keys=ON but skips WAL since in-memory doesn't need it -- actually in-memory ignores WAL pragma harmlessly, so it's fine to call the same function).

**src/db/migrate.ts** -- Migration runner:
- Use Umzug with better-sqlite3 directly (do NOT bring in Sequelize -- Umzug supports custom storage).
- Create a custom Umzug storage that uses a `_migrations` table in the same SQLite database to track which migrations have run. The storage needs `logMigration(name)`, `unlogMigration(name)`, and `executed()` methods.
- Configure Umzug to glob `src/db/migrations/*.ts` for migration files.
- Each migration exports `up(db: Database.Database)` and `down(db: Database.Database)`.
- Export `runMigrations(db: Database.Database)` that creates the Umzug instance and calls `umzug.up()`.
- Export `runMigrationsSync(db: Database.Database)` as a synchronous wrapper if possible (better-sqlite3 is sync, so migrations can be sync -- Umzug is async but we can await in the runner script).
- When this file is run directly (`tsx src/db/migrate.ts`), it should init a database at `./data/tasks.db` (creating the `data/` directory if needed) and run migrations.

**src/db/migrations/001-initial-schema.ts** -- Initial schema migration:
- Export `up(db)` that runs (all in a single transaction via `db.transaction()`):
  1. CREATE TABLE projects (id INTEGER PRIMARY KEY AUTOINCREMENT, name TEXT NOT NULL UNIQUE, description TEXT, created_at TEXT NOT NULL DEFAULT (datetime('now')), updated_at TEXT NOT NULL DEFAULT (datetime('now')))
  2. CREATE TABLE tasks (id INTEGER PRIMARY KEY AUTOINCREMENT, title TEXT NOT NULL, description TEXT, status TEXT NOT NULL DEFAULT 'open' CHECK(status IN ('open','in_progress','done','closed','blocked')), priority TEXT NOT NULL DEFAULT 'medium' CHECK(priority IN ('low','medium','high','urgent')), project_id INTEGER NOT NULL REFERENCES projects(id) ON DELETE CASCADE, assignee TEXT, created_by TEXT NOT NULL, due_date TEXT, created_at TEXT NOT NULL DEFAULT (datetime('now')), updated_at TEXT NOT NULL DEFAULT (datetime('now')))
  3. CREATE TABLE task_tags (id INTEGER PRIMARY KEY AUTOINCREMENT, task_id INTEGER NOT NULL REFERENCES tasks(id) ON DELETE CASCADE, tag TEXT NOT NULL, UNIQUE(task_id, tag))
  4. Indexes:
     - `CREATE INDEX idx_tasks_project_id ON tasks(project_id)`
     - `CREATE INDEX idx_tasks_project_status_assignee ON tasks(project_id, status, assignee)`
     - `CREATE INDEX idx_tasks_status_due_date ON tasks(status, due_date)`
     - `CREATE INDEX idx_tasks_assignee ON tasks(assignee)`
     - `CREATE INDEX idx_task_tags_task_id ON task_tags(task_id)`
     - `CREATE INDEX idx_task_tags_tag ON task_tags(tag)`
  5. FTS5 virtual table: `CREATE VIRTUAL TABLE tasks_fts USING fts5(title, description, content='tasks', content_rowid='id')`
  6. FTS5 sync triggers:
     - AFTER INSERT ON tasks: INSERT INTO tasks_fts(rowid, title, description) VALUES (new.id, new.title, new.description)
     - AFTER UPDATE ON tasks: INSERT INTO tasks_fts(tasks_fts, rowid, title, description) VALUES('delete', old.id, old.title, old.description); INSERT INTO tasks_fts(rowid, title, description) VALUES (new.id, new.title, new.description)
     - AFTER DELETE ON tasks: INSERT INTO tasks_fts(tasks_fts, rowid, title, description) VALUES('delete', old.id, old.title, old.description)
- Export `down(db)` that drops triggers, FTS table, task_tags, tasks, projects in reverse order.

IMPORTANT: For FTS5 content-sync tables, UPDATE triggers must use the delete-then-reinsert pattern (INSERT with 'delete' command first, then INSERT new values). This is the official SQLite FTS5 pattern for external content tables.

Do NOT use Sequelize at all. Umzug supports custom storage -- implement a simple SQLite-backed storage class directly with better-sqlite3.
