Implement the full WorkflowEngine class in `src/services/workflow-engine.ts`:

```typescript
export class WorkflowEngine {
  private currentDepth = 0;
  private static readonly MAX_CASCADE_DEPTH = 5;
  private unsubscribes: Array<() => void> = [];

  constructor(
    private readonly taskService: TaskService,
    private readonly taskRepo: ITaskRepository,
    private readonly eventBus: EventBus<any>
  ) {}

  start(): void {
    const unsub = this.eventBus.subscribe(
      'task.status_changed',
      (event) => this.handleStatusChanged(event)
    );
    this.unsubscribes.push(unsub);
  }

  stop(): void {
    for (const unsub of this.unsubscribes) unsub();
    this.unsubscribes = [];
  }
```

**handleStatusChanged logic:**
1. Check cascade depth: if `this.currentDepth >= MAX_CASCADE_DEPTH`, return immediately
2. Increment `this.currentDepth`
3. In a try/finally block (always decrement depth in finally):
   a. If event data has `parent_task_id` AND the new status is 'done':
      - Get all siblings: `taskRepo.findChildren(event.data.parent_task_id)`
      - Check if ALL siblings have status 'done'
      - If yes, get parent task via `taskRepo.findById(parent_task_id)`
      - Check parent status allows transition to 'done' (must be 'open' or 'in_progress' per VALID_STATUS_TRANSITIONS)
      - If valid, call `taskService.updateTask(parentId, { status: 'done' }, 'workflow')`
      - This emit triggers handleStatusChanged again synchronously (EventEmitter is sync)
      - The depth counter prevents infinite recursion

**Critical implementation detail:** EventEmitter.emit() in Node.js is SYNCHRONOUS. When taskService.updateTask emits task.status_changed, the EventBus calls this handler again WITHIN the same call stack. The `currentDepth` counter accurately tracks recursion depth because there is no async gap.

**Error handling:** Wrap the workflow logic in try/catch. Log errors but do not throw -- workflow failures should not crash the event handler or block other subscribers.
