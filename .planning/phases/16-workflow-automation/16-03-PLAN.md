**Transaction Atomicity (SC-5):**

The current WorkflowEngine calls taskService.updateTask for each cascading update. Each call is its own transaction (TaskRepository.update uses db.transaction internally). For crash safety, the ENTIRE cascade chain should be atomic.

However, there is a design tension: EventEmitter handlers run synchronously within TaskService.updateTask. The cascade IS the recursive EventBus triggering. Each individual update is already atomic. The risk is: if process crashes between update 1 and update 2 in a cascade, update 1 is committed but update 2 is not.

**Solution: Wrap the cascade entry point in a transaction.**

Modify WorkflowEngine to accept `db: Database.Database` as a constructor parameter. When handleStatusChanged is called at depth 0 (top-level entry), wrap the entire processing in `db.transaction(() => { ... })()`. Nested calls (depth > 0) run inside the same transaction because SQLite supports nested transaction semantics via better-sqlite3's nested transaction support (inner transactions become savepoints).

BUT: better-sqlite3 `db.transaction()` creates a function that, when called, wraps in BEGIN/COMMIT. Nested `db.transaction()` calls within an outer transaction automatically use savepoints. So the approach is:

1. Add `private db: Database.Database` to WorkflowEngine constructor
2. When `handleStatusChanged` is called at depth 0:
   ```typescript
   if (this.currentDepth === 0) {
     // Entry point — wrap entire cascade in one transaction
     const cascadeTx = this.db.transaction(() => {
       this.processCascade(event);
     });
     cascadeTx();
   } else {
     // Already inside a transaction (recursive call)
     this.processCascade(event);
   }
   ```
3. Move the actual logic (parent check, dependency check) into `processCascade(event)` private method
4. The depth tracking remains the same

**Important:** TaskRepository.update already uses `db.transaction()` internally. When called within an outer transaction, better-sqlite3 uses savepoints. If the outer transaction rolls back, all savepoints roll back too. This is the desired behavior.

**Update constructor calls** in index.ts, server.ts, and tests to pass `db` parameter.
