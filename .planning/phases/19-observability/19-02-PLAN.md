**server.ts changes:**

2. In the `Fastify({...})` constructor options, add two new properties:
   - `genReqId: () => randomUUID()` — generates UUID v4 for every request (replaces Fastify's default incrementing integer)
   - `requestIdHeader: false` — security: do not trust caller-supplied request IDs

   Place these right before the `logger:` property in the Fastify options object.

3. After the server is created and type provider is set (after `.withTypeProvider<ZodTypeProvider>()`), but before decorators, add an `onSend` hook:
   ```typescript
   // Stamp X-Request-ID on every response for client-side tracing
   server.addHook('onSend', async (request, reply) => {
     reply.header('X-Request-ID', request.id);
   });
   ```
   Place this immediately after the `server.setSerializerCompiler(serializerCompiler);` line.

Modify the `apiRequest` function in `src/cli/api/client.ts` to capture the `X-Request-ID` response header and surface it in two ways:

1. **On errors:** Include the request ID in `ApiClientError` for debugging.

   Add an optional `requestId` property to the `ApiClientError` class:
   ```typescript
   export class ApiClientError extends Error {
     public requestId?: string;
     constructor(
       message: string,
       public statusCode: number,
       public apiError: ApiErrorResponse,
       requestId?: string
     ) {
       super(message);
       this.name = 'ApiClientError';
       this.requestId = requestId;
     }
   }
   ```

2. **In the apiRequest function:** After `const response = await fetch(...)`, extract the request ID:
   ```typescript
   const requestId = response.headers.get('x-request-id') || undefined;
   ```

   Pass `requestId` to `ApiClientError` in the error handling block:
   ```typescript
   throw new ApiClientError(
     errorBody.message || `Request failed with status ${response.status}`,
     response.status,
     errorBody,
     requestId
   );
   ```

3. **Return request ID with successful responses:** Change the return type to include request ID metadata. Instead of returning raw `T`, create a small wrapper approach. However, to avoid breaking all 20+ callers, take a simpler approach:

   Add a module-level variable to store the last request ID:
   ```typescript
   let _lastRequestId: string | undefined;

   /** Get the request ID from the most recent API call */
   export function getLastRequestId(): string | undefined {
     return _lastRequestId;
   }
   ```

   In `apiRequest`, after extracting `requestId` from the response, set `_lastRequestId = requestId;`.

   This allows CLI commands to optionally read the request ID after API calls (for --json envelope or error display) without changing any existing function signatures.

Do NOT modify any existing CLI command files in this task — just make the request ID available. Future phases or the existing error handler in commands can use `getLastRequestId()` and `error.requestId` as needed.
