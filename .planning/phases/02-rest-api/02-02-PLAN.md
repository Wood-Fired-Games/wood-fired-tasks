**Create `src/api/hooks/error-handler.ts`:**
- Import `FastifyError`, `FastifyReply`, `FastifyRequest` from `fastify`
- Import `ValidationError`, `BusinessError`, `NotFoundError` from `../../services/errors.js`
- Export function `errorHandler(error: FastifyError | Error, request: FastifyRequest, reply: FastifyReply)`
- Log the error: `request.log.error(error)`
- Map errors to structured responses in this order:

  1. **ValidationError** (from Phase 1 service layer):
     ```typescript
     reply.code(400).send({
       error: 'VALIDATION_ERROR',
       message: 'Validation failed',
       details: error.fieldErrors,
     });
     ```

  2. **NotFoundError** (from Phase 1 service layer):
     ```typescript
     reply.code(404).send({
       error: 'NOT_FOUND',
       message: error.message,
       details: { entity: error.entity, id: error.id },
     });
     ```

  3. **BusinessError** (from Phase 1 service layer):
     ```typescript
     reply.code(422).send({
       error: 'BUSINESS_RULE_VIOLATION',
       message: error.message,
     });
     ```

  4. **Fastify validation errors** (from Zod schema validation via type provider -- these have a `statusCode` property and `code` like 'FST_ERR_VALIDATION'). Check with `'statusCode' in error`:
     ```typescript
     reply.code((error as FastifyError).statusCode || 400).send({
       error: (error as FastifyError).code || 'REQUEST_ERROR',
       message: error.message,
     });
     ```

  5. **Fallback** (unexpected errors -- NO stack traces in response):
     ```typescript
     reply.code(500).send({
       error: 'INTERNAL_ERROR',
       message: 'An unexpected error occurred',
     });
     ```

- IMPORTANT: The order matters. Check `instanceof ValidationError` BEFORE checking `'statusCode' in error` because Fastify wraps some errors and adds statusCode. Phase 1 custom errors should be caught first.

**Create `src/api/routes/health.ts`:**
- Export default async function (Fastify plugin signature) for the health check endpoint
- Define a single GET route at `/` (will be registered with prefix `/health` in server.ts):
  - No authentication required (this plugin is registered OUTSIDE the /api/v1 auth scope)
  - Schema: response 200 with Zod schema `z.object({ status: z.string(), timestamp: z.string(), version: z.string(), checks: z.object({ database: z.string() }) })`
  - Tags: ['health'], description: 'Service health check'
  - Handler:
    1. Try `fastify.db.prepare('SELECT 1').get()`
    2. On success: return `{ status: 'healthy', timestamp: new Date().toISOString(), version: '1.0.0', checks: { database: 'ok' } }`
    3. On failure: return reply.code(503).send `{ status: 'unhealthy', timestamp: ..., version: '1.0.0', checks: { database: 'failed' } }`
