**Create `src/api/routes/tasks/index.ts`:**
- Use `FastifyPluginAsyncZod` type from `fastify-type-provider-zod`
- Import `CreateTaskSchema`, `UpdateTaskSchema`, `TaskFiltersSchema` from `src/schemas/task.schema.ts`
- Import `TaskResponseSchema`, `TaskListResponseSchema`, `ErrorResponseSchema` from `./schemas.ts`
- Import `z` from `zod`

Define 5 route handlers:

1. **POST /** (Create task)
   - schema: body = CreateTaskSchema, response 201 = TaskResponseSchema
   - handler: `const task = fastify.taskService.createTask(request.body); return reply.code(201).send(task);`
   - Tags: ['tasks'], description: 'Create a new task'

2. **GET /** (List/filter tasks)
   - schema: querystring = TaskFiltersSchema, response 200 = TaskListResponseSchema
   - handler: `const tasks = fastify.taskService.listTasks(request.query); return reply.send(tasks);`
   - Note: querystring params will come as strings. For project_id, use `z.coerce.number()` in a querystring-specific schema OR handle coercion. The existing TaskFiltersSchema has `z.number().int().positive()` for project_id -- this will fail for query strings which arrive as strings. Create a `QueryTaskFiltersSchema` that wraps TaskFiltersSchema fields with coercion where needed: `project_id: z.coerce.number().int().positive()` for the querystring. Other string fields are fine. For the `tags` field in querystring, accept comma-separated string and transform: `tags: z.string().transform(s => s.split(',')).optional()` or accept `z.array(z.string())` (Fastify handles repeated query params as arrays).
   - Tags: ['tasks'], description: 'List tasks with optional filters'

3. **GET /:id** (Get task)
   - schema: params = `z.object({ id: z.coerce.number().int().positive() })`, response 200 = TaskResponseSchema
   - handler: `const task = fastify.taskService.getTask(request.params.id); return reply.send(task);`
   - Tags: ['tasks'], description: 'Get task by ID'

4. **PUT /:id** (Update task)
   - schema: params = `z.object({ id: z.coerce.number().int().positive() })`, body = UpdateTaskSchema, response 200 = TaskResponseSchema
   - handler: `const task = fastify.taskService.updateTask(request.params.id, request.body); return reply.send(task);`
   - Tags: ['tasks'], description: 'Update task by ID'

5. **DELETE /:id** (Delete task)
   - schema: params = `z.object({ id: z.coerce.number().int().positive() })`, response 204 = `z.null().describe('No content')`
   - handler: `fastify.taskService.deleteTask(request.params.id); return reply.code(204).send();`
   - Tags: ['tasks'], description: 'Delete task by ID'

All route handlers call Phase 1 service methods directly. Errors thrown by services (ValidationError, NotFoundError, BusinessError) will bubble up to Fastify's default error handler for now -- plan 02-02 adds the custom error handler. For this plan, the routes should still work because unhandled errors produce 500s, and tests can verify the happy paths plus auth. Error mapping tests come in plan 02-02.
